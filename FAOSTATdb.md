Your specifications are coherent and useful. The central idea—**a reproducible local DuckDB mirror of FAOSTAT bulk files with normalized dimensions and minimal semantic alteration**—is strong. I would, however, tighten a few design choices early, because they affect the package architecture substantially.

## Overall assessment

`FAOSTATdb` is a good name and communicates the purpose clearly. I would position it not as an analytical harmonization package, but as a **local, queryable FAOSTAT bulk-data mirror**. That distinction matters: existing tools such as the R `FAOSTAT` package focus on downloading and reading FAOSTAT data, sometimes with harmonization utilities, whereas your proposal is closer to a durable database build system with strict preservation of source information. The FAOSTAT bulk metadata already gives exactly the kind of inventory needed for such a tool: dataset code, dataset name, update date, compression format, file type, file size, row count, and download URL. ([bulks-faostat.fao.org][1])

I would make the project’s principle explicit:

> FAOSTATdb should preserve the statistical content of FAOSTAT exactly, while removing storage-level duplication and adding reproducibility metadata.

That principle gives clear guidance on all borderline cases.


## Data model: good idea, but use a hybrid normalization strategy

Your instinct to avoid repeated labels in fact tables is right. FAOSTAT normalized CSVs tend to repeat area labels, item labels, element labels, unit labels, flag labels, etc. Storing those labels once in dimension tables will reduce size and make joins cleaner.

I would propose the following structure.

### Core metadata tables

```text
faostat_dataset
- dataset_code
- dataset_name
- topic
- dataset_description
- contact
- email
- date_update
- compression_format
- file_type
- file_size_raw
- file_rows_declared
- file_location
- downloaded_at
- source_metadata_url
- source_metadata_hash
- archive_sha256
- import_status
```

The metadata JSON is valuable enough to store in full, not only as parsed fields. It contains official update dates and download locations; for example, recent datasets in the current metadata include `DateUpdate`, `FileSize`, `FileRows`, and `FileLocation`. ([bulks-faostat.fao.org][1])

I would add:

```text
faostat_build
- build_id
- started_at
- completed_at
- faostatdb_version
- duckdb_version
- python_version
- os
- metadata_snapshot_sha256
- command_line
- config_sha256
```

This makes the database auditable.

### Fact tables

There are two plausible designs:

#### Option A: one fact table per FAOSTAT dataset

Example:

```text
data_qcl
data_fbs
data_cp
data_et
```

Advantages:

- Closest to source files.
- Easier to preserve odd dataset-specific columns.
- Faster imports.
- Lower risk of coercing heterogeneous datasets into a false common schema.

Disadvantages:

- Queries across datasets require dynamic table names or views.
- More tables.

#### Option B: one large unified fact table

Example:

```text
faostat_data
- dataset_code
- area_code
- item_code
- element_code
- year
- unit_code
- value
- flag_code
- ...
```

Advantages:

- Uniform access.
- Easy cross-domain search.

Disadvantages:

- FAOSTAT domains are not perfectly homogeneous.
- Some datasets have monthly, seasonal, donor, recipient, purpose, sex, age, source, or other dataset-specific dimensions.
- You risk creating many sparse columns or losing original structure.

My recommendation: **use Option A as the canonical storage model**, and optionally create a convenience view for datasets that share the classic FAOSTAT shape.

```sql
CREATE VIEW faostat_data_long AS
SELECT 'QCL' AS dataset_code, ...
FROM data_qcl
UNION ALL
SELECT 'FBS' AS dataset_code, ...
FROM data_fbs;
```

This avoids over-normalizing heterogeneous data.

## Do not remove all labels blindly

I agree that labels should not be repeatedly stored in fact tables. But I would preserve the **original source column names and codes**. For example, if FAOSTAT contains both `Area Code` and `Area`, the fact table should probably retain only `area_code`, while `area` goes into `dim_area`. However, I would also keep a raw-import audit option:

```text
--keep-raw-tables
```

This could store untouched imported CSVs as DuckDB tables under a `raw_` prefix. It is not necessary by default, but invaluable for debugging regressions and validating losslessness.

A good compromise:

- Default database: normalized, no repeated labels.
- Optional debug database: normalized tables plus raw tables.
- Always retain archive hashes and column mappings.

## Flags: preserve them fully

You are right to insist on retaining flags. I would go further: preserve both **flag codes and flag descriptions**.

The R `FAOSTAT` package documentation notes that FAOSTAT bulk ZIP archives may contain metadata files about flags and symbols. ([RDocumentation][2]) This suggests that `FAOSTATdb` should not treat flags as secondary information.

Recommended tables:

```text
dim_flag
- flag_code
- flag_description
- source_dataset_code nullable
- source_file
```

Fact tables should retain:

```text
flag_code
note_code    -- if present
symbol_code  -- if present
```

Do not collapse flags to a single boolean such as `is_estimated`. That would lose information.

## Country metadata: useful, but separate source-derived from package-derived

Adding information such as whether an area is a country or region, and the years of existence of countries, is valuable. But this is no longer pure FAOSTAT ingestion. I would separate it carefully:

```text
dim_area
- area_code
- area_m49_code
- area_label
- area_label_fao_original

area_classification
- area_code
- is_country
- is_region
- is_aggregate
- classification_source
- valid_from
- valid_to
- confidence
```

The key point is that **country-existence metadata should be explicitly sourced**. FAOSTAT covers countries, territories, and FAO regional groupings; the public FAOSTAT page describes coverage of over 245 countries and territories and FAO regional groupings from 1961 onward. ([FAOHome][3]) But historical country validity is a delicate issue. Soviet Union, Yugoslavia, Czechoslovakia, Serbia and Montenegro, Sudan/South Sudan, Ethiopia/Eritrea, and China-related codes require careful handling.

I would make this an optional enrichment layer:

```bash
faostatdb build --enrich-areas
```

or in config:

```toml
[enrichment]
area_classification = true
historical_validity = true
```

The default should remain source-faithful.

## Configuration: use TOML, not YAML

You suggested YAML. I would push back mildly.

YAML is convenient, but it requires an additional dependency in Python. If minimal dependencies are important, **TOML is better**, because Python 3.11+ includes `tomllib` in the standard library for reading TOML.

Example:

```toml
[build]
database = "faostat.duckdb"
download_dir = ""
keep_archives = false
jobs = 6
overwrite = false

[datasets]
include = [
  "QCL",
  "FBS",
  "CP",
  "ET",
]

exclude = [
  "FA",
  "CBH",
]
```

For your “comment unwanted data” workflow, TOML works well:

```toml
[datasets]
include = [
  "QCL", # Crops and livestock products
  "FBS", # Food balances
  # "FA",  # Discontinued: Food Aid Shipments
]
```

I would support both styles:

```toml
[datasets]
mode = "all"        # all | include | exclude
include = []
exclude = ["FA", "CBH"]
```

This is less error-prone than asking users to comment hundreds of lines.


## Dependencies: avoid external `zip`; use Python’s standard library

You say `duckdb` and `zip` are unavoidable. I would challenge the `zip` part.

Python’s standard library has `zipfile`, including integrity testing via `ZipFile.testzip()`. That gives cross-platform archive validation without relying on a system `zip` executable, which may not be present on Windows.

So the minimal dependency set could be:

```text
Required:
- Python >= 3.11
- duckdb

Optional:
- rich, for modern progress display
- platformdirs, for OS-appropriate cache directories
```

I would avoid making `rich` required at first. You can implement a basic progress display with the standard library, then add `rich` as an optional extra:

```bash
pipx install "faostatdb[ui]"
```

That said, for the modern package-manager-style UI you want, `rich` is clearly the best ergonomic choice.

## Downloading and restart logic: your requirements are excellent

The hot-restart requirement is important and should be designed from the beginning.

I would implement a small download state machine:

```text
pending
downloading
downloaded
zip_valid
zip_invalid
importing
imported
failed
```

Store it in a local manifest, for example:

```text
.faostatdb-downloads/manifest.sqlite
```

or simpler:

```text
.faostatdb-downloads/manifest.jsonl
```

Each record:

```json
{
  "dataset_code": "QCL",
  "url": "...",
  "expected_size": "...",
  "expected_rows": 123456,
  "archive_path": "...",
  "sha256": "...",
  "status": "zip_valid",
  "attempts": 2,
  "last_error": null,
  "downloaded_at": "2026-06-24T..."
}
```

### Integrity checks

For each ZIP:

1. Download to `filename.part`.
2. Rename atomically to `filename.zip` only after completion.
3. Run `zipfile.ZipFile(path).testzip()`.
4. Optionally compare file size against metadata.
5. Optionally count rows during import and compare to `FileRows`.

The metadata JSON contains declared row counts and file sizes, so the package can perform useful post-import validation. ([bulks-faostat.fao.org][1])

### Retry policy

Use exponential backoff with bounded retries:

```text
max_retries = 3
backoff = 2, 5, 15 seconds
```

After that, mark the dataset failed and continue unless the user requested strict mode:

```bash
faostatdb build --strict
```

In strict mode, any failed dataset fails the whole build.


## Where to download files

Do not default blindly to `/tmp`.

Large FAOSTAT builds may exceed temporary partition limits, especially on Linux systems where `/tmp` may be memory-backed or small. Also, Windows and macOS have different cache conventions.

Recommended behavior:

1. If `--download-dir` is given, use it.

2. Else if `keep_archives = true`, use a project-local folder:

   ```text
   ./faostatdb_archives/
   ```

3. Else use an OS cache directory:

   ```text
   Linux:   ~/.cache/faostatdb/
   macOS:   ~/Library/Caches/faostatdb/
   Windows: %LOCALAPPDATA%\faostatdb\Cache\
   ```

If avoiding the `platformdirs` dependency, implement a small internal resolver.

For hot restart, even when `keep_archives = false`, I would not delete archives until the database build succeeds. On success:

```text
if keep_archives = false:
    delete valid archives
else:
    keep archives and manifest
```

If interrupted, keep them.

## Parallel downloads: good, but make concurrency adaptive

Parallel downloading should be included. But I would avoid hard-coding an “optimal” number. Network, disk, FAO server behavior, and geography matter.

Default:

```text
jobs = min(8, 2 * cpu_count)
```

But allow:

```bash
faostatdb build --jobs 4
```

In documentation, report benchmarks on a few connections, but phrase them as indicative rather than universal.

Also separate:

```toml
[performance]
download_jobs = 6
import_threads = 4
duckdb_memory_limit = "8GB"
```

DuckDB import performance depends on memory and CPU; network concurrency and database import parallelism should not necessarily be the same.

## Database import: avoid loading everything through pandas

A crucial design point: do **not** import via pandas. That would create unnecessary memory pressure and add a dependency.

Use DuckDB’s CSV reader directly where possible. Since the source files are ZIP archives, there are two robust paths:

1. Extract each CSV to a temporary directory, then use DuckDB `read_csv`.
2. Stream extracted CSV content through Python into DuckDB.

I would choose extraction to a temporary build directory because it is simpler, faster, and easier to validate. Delete extracted CSVs immediately after import unless debugging.

For each dataset:

```sql
CREATE TABLE data_qcl AS
SELECT ...
FROM read_csv('.../QCL.csv', header = true, auto_detect = true);
```

Then create dimension tables from distinct code-label pairs.

A complication: labels may not be globally unique across all datasets. For example, `Item Code` may mean different things in different domains. Therefore, dimension tables should usually be keyed by both dimension type and dataset family, or by a global namespace.

Example:

```text
dim_item
- dataset_code
- item_code
- item_label
```

rather than assuming `item_code` is globally meaningful.

For dimensions that are plausibly global, such as areas and flags, you can later consolidate.

```sql
.open faostat.duckdb

SELECT
    a.area_label,
    y.year,
    d.value,
    u.unit_label
FROM data_qcl AS d
LEFT JOIN dim_area AS a
    ON d.area_code = a.area_code
LEFT JOIN dim_unit AS u
    ON d.unit_code = u.unit_code
WHERE d.item_code = '15'
  AND d.element_code = '5510'
  AND y.year >= 2000;
```

The exact query will depend on final schema, but the README should show joins to dimensions explicitly.

### Query in R

```r
library(DBI)
library(duckdb)

con <- dbConnect(duckdb(), "faostat.duckdb", read_only = TRUE)

  FROM faostat_dataset
  GROUP BY dataset_code

dbDisconnect(con, shutdown = TRUE)
```

```python
import duckdb

con = duckdb.connect("faostat.duckdb", read_only=True)
```

## Proposed MVP

I would define the first release narrowly:

### Version 0.1

- Fetch `datasets_E.json`.
- Let user select all/include/exclude datasets.
- Download ZIPs in parallel.
- Validate ZIPs.
- Import each dataset into one DuckDB table per dataset.
- Store source metadata and build metadata.
- Normalize column names.
- Preserve flags.
- Delete archives after successful build unless requested.
- Support hot restart.
- Provide README examples for R, Python, Julia, DuckDB.
- CI on Linux/macOS/Windows.

### Version 0.2

- Create dimension tables.
- Remove repeated labels from fact tables.
- Add column-mapping tables.
- Add validation against declared row counts.
- Add `faostatdb info` and `faostatdb validate`.

### Version 0.3

- Area enrichment.
- Historical country-validity metadata.
- Rich progress UI.
- Optional raw tables.
- Benchmarking of concurrency.

This staging reduces risk. The dimension extraction step is where many edge cases will appear, so I would not block the first usable version on perfect normalization.

## Main design decisions to settle early

I would decide these before coding:

1. **Canonical storage model**: one fact table per dataset, with optional unified views.
2. **Config format**: TOML rather than YAML.
3. **Language**: Python is the pragmatic choice because of DuckDB bindings, cross-platform support, `zipfile`, `tomllib`, and easy CLI packaging.
4. **Dependency policy**: required `duckdb` only; optional `rich` and perhaps `platformdirs`.
5. **Archive handling**: validate with Python `zipfile`, not external `zip -T`.
6. **Reproducibility metadata**: store metadata JSON hash, archive hashes, download dates, and package versions.
7. **Area enrichment**: optional, clearly distinguished from source FAOSTAT content.
8. **Default full build**: either require confirmation or require `--all`/`--yes`.

## R: use `duckplyr` / `dplyr` as the main example

This is exactly the right direction. `duckplyr` is designed as a DuckDB-backed version of `dplyr`; its documentation describes it as a drop-in replacement that runs existing `dplyr` code using DuckDB where possible. ([duckplyr.tidyverse.org][1])

A README example could look like this:

```r
library(duckplyr)
library(dplyr)

# Connect to the FAOSTATdb database
con <- DBI::dbConnect(duckdb::duckdb(), "faostat.duckdb", read_only = TRUE)

# Reference DuckDB tables lazily
crops <- tbl(con, "data_qcl")
areas <- tbl(con, "dim_area")
items <- tbl(con, "dim_item")
elements <- tbl(con, "dim_element")
units <- tbl(con, "dim_unit")

# Example: wheat production in France since 2000
wheat_france <- crops |>
  filter(
    area_code == "250",
    item_code == "15",
    element_code == "5510",
    year >= 2000
  ) |>
  left_join(areas, by = "area_code") |>
  left_join(items, by = c("dataset_code", "item_code")) |>
  left_join(elements, by = c("dataset_code", "element_code")) |>
  left_join(units, by = "unit_code") |>
  select(
    area = area_label,
    item = item_label,
    element = element_label,
    year,
    value,
    unit = unit_label,
    flag = flag_code
  ) |>
  arrange(year) |>
  collect()

wheat_france
```

This is much more appropriate for FAOSTAT users. The user does not need to know SQL, but still gets lazy execution against DuckDB.

I would also include a plotting example:

```r
library(ggplot2)

wheat_france |>
  ggplot(aes(x = year, y = value)) +
  geom_line() +
  labs(
    title = "Wheat production in France",
    x = NULL,
    y = unique(wheat_france$unit)
  )
```

A more discoverable example would avoid assuming users know FAOSTAT item codes:

```r
items |>
  filter(grepl("Wheat", item_label, ignore.case = TRUE)) |>
  select(dataset_code, item_code, item_label) |>
  collect()
```

That is essential. Most users will search labels first, then use codes.

## Python: prefer Ibis as the closest equivalent to `dplyr`

For Python, I would not make raw `duckdb.sql()` the main example. I would use **Ibis** as the primary non-SQL interface. Ibis provides a dataframe-like expression API across backends including DuckDB, and its own materials describe this as an API alternative to SQL for analytical engines. ([Ibis][2])

Example:

```python
import ibis

con = ibis.duckdb.connect("faostat.duckdb", read_only=True)

crops = con.table("data_qcl")
areas = con.table("dim_area")
items = con.table("dim_item")
elements = con.table("dim_element")
units = con.table("dim_unit")

wheat_france = (
    crops
    .filter(
        (crops.area_code == "250")
        & (crops.item_code == "15")
        & (crops.element_code == "5510")
        & (crops.year >= 2000)
    )
    .left_join(areas, crops.area_code == areas.area_code)
    .left_join(
        items,
        (crops.dataset_code == items.dataset_code)
        & (crops.item_code == items.item_code),
    )
    .left_join(
        elements,
        (crops.dataset_code == elements.dataset_code)
        & (crops.element_code == elements.element_code),
    )
    .left_join(units, crops.unit_code == units.unit_code)
    .select(
        area=areas.area_label,
        item=items.item_label,
        element=elements.element_label,
        year=crops.year,
        value=crops.value,
        unit=units.unit_label,
        flag=crops.flag_code,
    )
    .order_by(crops.year)
)

df = wheat_france.execute()
print(df)
```

Then a discovery example:

```python
items = con.table("dim_item")

wheat_codes = (
    items
    .filter(items.item_label.re_search("(?i)wheat"))
    .select("dataset_code", "item_code", "item_label")
    .execute()
)

print(wheat_codes)
```

A second Python example could use Polars if the database exposes views or if the result is first pulled into a lazy frame, but Ibis is probably the cleanest general-purpose Python recommendation for DuckDB-backed querying.

## Julia: be more careful

Julia is less straightforward if the goal is “no SQL.” The official DuckDB Julia client supports querying Julia DataFrames and executing through DuckDB, but the canonical interaction is still closer to DuckDB queries than to a mature `dplyr`-like translation layer. ([DuckDB][3])

For Julia, I would probably document two options:

### Option 1: beginner-friendly, materialize a filtered slice, then use DataFrames

This is easier for users who know DataFrames.jl:

```julia
using DuckDB
using DBInterface
using DataFrames
using DataFramesMeta

con = DBInterface.connect(DuckDB.DB, "faostat.duckdb")

# Load a relevant slice from DuckDB
crops = DataFrame(DBInterface.execute(con, """
    SELECT *
    FROM data_qcl
    WHERE area_code = '250'
      AND item_code = '15'
      AND element_code = '5510'
      AND year >= 2000
"""))

areas = DataFrame(DBInterface.execute(con, "SELECT * FROM dim_area"))
units = DataFrame(DBInterface.execute(con, "SELECT * FROM dim_unit"))

wheat_france =
    crops |>
    x -> leftjoin(x, areas, on = :area_code) |>
    x -> leftjoin(x, units, on = :unit_code) |>
    x -> select(
        x,
        :area_label => :area,
        :year,
        :value,
        :unit_label => :unit,
        :flag_code => :flag
    )

wheat_france
```

This still uses a small amount of SQL to load the initial slice, which is not ideal, but it avoids asking users to write complex joins in SQL.

### Option 2: use a package-specific helper function

A stronger design would be to make `FAOSTATdb` expose a few convenience views or helper metadata files so Julia users can do:

```julia
using DuckDB
using DataFrames

con = DBInterface.connect(DuckDB.DB, "faostat.duckdb")

qcl = faostat_table(con, "QCL")

wheat_france =
    qcl |>
    x -> filter(row ->
        row.area_code == "250" &&
        row.item_code == "15" &&
        row.element_code == "5510" &&
        row.year >= 2000,
        x
    )
```

But this requires a Julia helper package, which is probably outside the scope of the first release.

So my recommendation is: **R and Python should have genuinely non-SQL examples; Julia should have a pragmatic mixed example unless you want to maintain a small Julia wrapper.**

## Schema implication: provide “labelled views”

Your point has an important consequence for schema design. If users are not writing SQL, joins should be as painless as possible.

I would keep normalized fact tables as the canonical storage, but also generate **labelled convenience views**:

```text
data_qcl              -- compact canonical fact table
view_qcl_labelled     -- fact table joined to area/item/element/unit/flag labels
```

For example:

```text
view_qcl_labelled
- dataset_code
- area_code
- area_label
- item_code
- item_label
- element_code
- element_label
- year
- unit_code
- unit_label
- value
- flag_code
- flag_label
```

Then R becomes much simpler:

```r
library(duckplyr)
library(dplyr)

con <- DBI::dbConnect(duckdb::duckdb(), "faostat.duckdb", read_only = TRUE)

qcl <- tbl(con, "view_qcl_labelled")

wheat_france <- qcl |>
  filter(
    area_label == "France",
    grepl("Wheat", item_label, ignore.case = TRUE),
    element_label == "Production",
    year >= 2000
  ) |>
  select(area_label, item_label, element_label, year, value, unit_label, flag_code) |>
  arrange(year) |>
  collect()
```

That is much closer to how a non-SQL FAOSTAT user thinks.

The storage cost of views is negligible if they are not materialized. I would therefore make these labelled views part of the default build.
